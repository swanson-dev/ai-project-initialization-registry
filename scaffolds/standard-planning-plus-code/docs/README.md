# Documentation Structure Rules

This scaffold is planning-first. Documentation is organized by numbered stages to preserve order and traceability.

## Folder rules

- Place planning, specs, and governance docs in the appropriate numbered folder.
- Keep implementation code out of `/docs`.
- Keep implementation code in `/app` only.
- Maintain one source-of-truth file per topic when possible.

## Folders

- `00-overview`: project summary and context
- `01-requirements`: requirements and acceptance criteria
- `02-architecture`: architecture design and constraints
- `03-ui-ux`: UI/UX artifacts and approval notes
- `04-project-management`: planning and execution tracking
- `05-research`: external findings and evaluations
- `06-decisions`: ADR-style decision records
- `07-status`: progress reports and changelogs
- `08-testing-quality`: test strategy and quality reports
- `09-release`: release readiness and rollout notes
- `10-implementation`: implementation plans and mappings
